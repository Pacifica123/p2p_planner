# Доменные инварианты

## Общие

1. Каждая основная sync-visible доменная сущность имеет UUIDv7.
2. Любая сущность, участвующая в синхронизации, должна иметь стабильный canonical ID.
3. Любое удаление, которое должно быть донесено до другой реплики, не может
   моделироваться только физическим удалением строки.
4. `deleted_at` и/или tombstone должны трактоваться как доменный факт удаления.
5. Future-ready сущности не должны подменять или раздувать обязательный MVP scope.

## Workspace

1. Board всегда принадлежит одному workspace.
2. User не может иметь две активные membership-записи в одном и том же workspace.
3. Owner workspace не может быть удален без передачи владения или архивации workspace.

## Board и Column

1. Column всегда принадлежит одной board.
2. Порядок колонок должен быть явно представлен отдельным полем позиции.
3. Количество колонок не ограничивается заранее фиксированной схемой.
4. Названия колонок уникальны только в пределах одной доски по нормализованному виду,
   если это правило не будет отдельно ослаблено.
5. Архивированная доска не должна принимать обычные пользовательские изменения.

## Card

1. Card принадлежит одной board и в текущий момент находится максимум в одной column.
2. Порядок карточек внутри колонки выражается отдельным позиционным полем.
3. Завершенная карточка должна иметь либо `completed_at`, либо статус,
   трактуемый как completed, согласно перечислению домена.
4. Удаленная карточка не должна продолжать участвовать в reorder-операциях.
5. В MVP card не предполагает обязательного существования assignees/watchers/custom fields.

## Checklist

1. ChecklistItem не может существовать без Checklist.
2. Порядок элементов чеклиста должен быть независим от времени обновления.
3. Отметка выполнения элемента не должна приводить к потере истории изменения.

## Labels

1. Label определяется в рамках board.
2. Связь card-label не должна ссылаться на удаленную или чужую board label.

## Appearance

1. Настройки внешнего вида доски не должны ломать читаемость интерфейса.
2. Отсутствие theme/wallpaper должно приводить к безопасным значениям по умолчанию.
3. Appearance-сущности считаются future-ready и не обязательны для MVP UI.

## Sync

1. Каждое change-событие создается одной репликой.
2. Один event ID не может описывать разные payload.
3. Повторная доставка одного и того же события должна быть идемпотентной.
4. Tombstone имеет приоритет над устаревшим snapshot/patch без доказательства
   более нового допустимого восстановления.
5. SyncCursor обновляется только вперед относительно известного прогресса.

## Integrations

1. Интеграции не должны ломать core-домен при отключении внешнего провайдера.
2. IntegrationLink не должен существовать без IntegrationConnection.
3. Ошибка внешнего провайдера не должна повреждать локальное состояние workspace.
4. Integration-сущности не являются обязательной частью MVP.
