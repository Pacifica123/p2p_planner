# Права доступа

Документ фиксирует базовую модель прав. Это не окончательная RBAC/ABAC-схема,
а стартовый контракт для backend и frontend.

## Уровни доступа

### 1. Workspace-level роли
Это **канонический уровень ролей для MVP**:
- `owner`
- `admin`
- `member`
- `viewer`

### 2. Потенциальные board-level overrides
На первом этапе **не входят в MVP**, но архитектура может допускать:
- board_admin
- board_editor
- board_commenter
- board_viewer

## Базовые правила

### Owner
Может:
- управлять workspace;
- удалять и архивировать доски;
- управлять участниками;
- менять глобальные настройки workspace;
- передавать владение.

### Admin
Может:
- создавать доски;
- редактировать доски;
- управлять участниками в рамках разрешений owner;
- видеть служебные статусы синхронизации.

### Member
Может:
- создавать и редактировать карточки;
- создавать колонки и доски, если это разрешено настройками workspace;
- комментировать;
- работать с чеклистами и метками в пределах выданных прав.

### Viewer
Может:
- читать содержимое досок;
- видеть комментарии;
- не может изменять бизнес-состояние.

## Нефункциональные требования к модели прав

1. Права должны быть вычислимы как на backend, так и на frontend.
2. Нельзя зашивать права только в UI.
3. Проверки прав должны быть независимы от транспортного слоя.
4. Sync-событие, нарушающее доменные правила доступа, может быть:
   - отклонено;
   - помечено как конфликтное;
   - помещено в quarantine/invalid state, если такая модель будет введена.

## Что сознательно не включено в MVP

Не являются обязательной частью v1:
- приглашения по ссылке;
- временные роли;
- read-only public boards;
- board-specific permissions;
- service accounts/integration actors.


## Режимы guest / private / shared / public

### Guest
- не имеет server-side аккаунта;
- может существовать только как local-only режим клиента;
- не может выполнять server-side write операции;
- не участвует в membership и sync backend flow.

### Private
- основной режим MVP по умолчанию;
- доступ только у участников workspace;
- backend всегда требует authentication и membership check.

### Shared
- все еще требует authentication;
- не делает workspace публичным для интернета;
- отличается от private продуктовой семантикой совместной работы, а не ослаблением auth.

### Public readonly
- schema-ready, но не обязательный baseline MVP режим;
- если включается позже, то дает только anonymous read;
- anonymous write и anonymous membership management запрещены.
