# Термины и границы понятий

Документ фиксирует **канонические термины**, чтобы одинаковые слова не означали
разные вещи в product, domain, backend и sync-документах.

## Продуктовые термины

### Workspace
Верхнеуровневая область данных и доступа. Объединяет участников, доски и
настройки. Это основной контейнер совместной работы.

### Workspace member
Связь пользователя с workspace. В MVP роль задается на уровне workspace.

### Board
Kanban-доска внутри workspace. Это основная рабочая поверхность v1.

### Column
Колонка внутри board. Колонки принадлежат конкретной доске и имеют явный порядок.

### Card
Основная рабочая единица на доске. В v1 card содержит core-поля, labels,
checklists и comments.

### Label
Метка уровня board, которую можно навешивать на card.

### Checklist / Checklist item
Набор подзадач внутри card и отдельный пункт этого набора.

### Comment
Комментарий к card. В MVP это простой текстовый комментарий без rich editor
как обязательного требования.


### Activity entry
Пользовательская запись истории изменений, пригодная для показа на экране board
или card. Это не sync-event и не технический audit.

### History
User-facing хронологический список `activity entry` в рамках конкретной board или
card. History нужна для понимания контекста изменений, а не для восстановления
всей модели данных.

### Audit log
Технический server-side журнал действий и запросов. Он нужен для admin/security/
diagnostic use-cases и не тождественен обычной пользовательской истории.

### User appearance preferences
Персональные настройки внешнего вида клиента: app theme, density, reduce motion.

### Board appearance settings
Общие для доски настройки внешнего вида: theme preset, wallpaper config и display toggles.

### Wallpaper config
Пара `kind + value`, описывающая фон доски без обязательной привязки к файловому asset.

### User
Глобальная серверная identity. Именно user является субъектом владения,
membership и аутентификации.

### Device
Учетная запись клиентского устройства или установки, привязанная к user.
Device не равен session и не равен replica.

### Session
Серверная запись активного login-сеанса. Session связана с user и опционально с
конкретным device.

### Replica
Sync-идентичность конкретного клиентского экземпляра. Replica не подменяет
user/session, а дополняет их для sync и idempotency.

## Состояния и операции

### Archive
Перевод сущности в неактивное состояние **без утверждения, что ее никогда не было**.
Архивированная сущность обычно скрывается из обычного рабочего потока, но не
равна удаленной.

### Delete
Удаление сущности из прикладного состояния. В sync-aware модели факт удаления
должен быть воспроизводим между репликами.

### Tombstone
Специальная запись или доменный факт, подтверждающий удаление для других реплик.
Tombstone — это не просто UI-статус, а sync-техническое представление удаления.

### Complete / completed state
Состояние выполненной card или checklist item. Оно не тождественно архивированию.

### Reorder
Отдельная операция изменения порядка колонок, карточек или checklist items.
Порядок нельзя выводить только из `updated_at`.

## Архитектурные термины

### Local-first
Подход, при котором клиент хранит локально рабочее состояние и может переживать
сетевую нестабильность без разрушения UX.

Для **этого проекта в MVP** local-first не означает backendless или full p2p.

### Web-first
Первый доводимый клиент — web. Именно web определяет границы MVP.

### Sync-ready
Архитектура уже учитывает реплики, change log, cursors, tombstones и conflict
policy, даже если полный sync UX еще не выведен наружу.

### Future-ready
Элемент модели, границы модуля или сущность предусмотрены заранее, но не входят
в обязательный объем первой версии.


### Guest mode
Анонимный local-only режим клиента без server-side user/session/device.

### Shared access
Режим совместной работы, который все равно требует authentication. Shared не
означает public internet access.

### Public readonly
Опциональный режим анонимного read-only доступа. Для baseline MVP считается
future-ready и по умолчанию выключен.

## Что считаем спорными и уже зафиксированными трактовками

1. **Local-first != backendless в MVP.**
2. **Optional p2p != обязательный transport первого релиза.**
3. **Archive != delete.**
4. **Completed != archived.**
5. **Activity history != audit log.**
6. **Sync-ready foundation != полный пользовательский sync surface.**
7. **Future-ready сущность != обещанная MVP-фича.**
