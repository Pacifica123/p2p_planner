# Протокол синхронизации

Этот документ описывает базовую семантику обмена изменениями между репликами.
Это **не wire-level RFC для полного p2p первого релиза**, а проектный контракт,
который задает общую sync-семантику и future-ready транспортные формы.

## Цели

1. Обмен изменениями между репликами.
2. Идемпотентная повторная доставка событий.
3. Поддержка локальных изменений и последующей синхронизации.
4. Поддержка удаления через tombstones.
5. Возможность частичной синхронизации и курсоров прогресса.

## Что входит в MVP

В MVP обязателен только backend-coordinated sync слой для web-клиента:
- клиент имеет локальное состояние и pending changes;
- backend принимает и подтверждает изменения;
- модель знает о `replica`, `cursor`, `change event` и `tombstone`;
- конфликтная политика для core сущностей детерминирована.

## Что не надо читать как обязательство MVP

Ниже описаны и будущие транспортные формы. Это не означает, что в первом релизе
уже реализуются:
- peer-to-peer transport;
- relay/bootstrap flows;
- snapshot exchange как пользовательский сценарий;
- полный набор wire-level sync message types.

## Базовые сущности протокола

### Replica
Источник и получатель sync-потока.

### Envelope
Транспортный контейнер для одного sync-сообщения.
В MVP может быть выражен как HTTP request/response exchange, а не как отдельный p2p wire format.

### ChangeEvent
Отдельное изменение по сущности.

### SnapshotManifest
Метаданные для пакетного snapshot/backup exchange.
Future-ready capability.

### SyncCursor
Маркер того, до какого места одна реплика знает журнал другой.

## Версии

Протокол имеет поле:
- `protocolVersion`

Несовместимые изменения протокола требуют повышения версии.

## Типы сообщений

### 1. hello
Используется для:
- объявления реплики;
- согласования версии;
- передачи метаданных о возможностях.

**Статус:** future-ready / необязателен как отдельный wire message в MVP.

### 2. pull_request
Запросить изменения после некоторого курсора.

**Статус:** семантически нужен; конкретная форма в MVP может быть HTTP endpoint.

### 3. push_events
Передать пачку событий.

**Статус:** семантически нужен; конкретная форма в MVP может быть HTTP endpoint.

### 4. ack
Подтверждение приема и прогресса.

**Статус:** нужен как семантика подтверждения.

### 5. snapshot_offer
Предложение передать snapshot.

**Статус:** post-MVP / future-ready.

### 6. snapshot_chunk
Передача части snapshot.

**Статус:** post-MVP / future-ready.

### 7. conflict_notice
Сообщение о конфликте или отклонении события.

**Статус:** допускается как семантика; отдельный UX может появиться позже.

## Минимальный сценарий обмена

1. Реплика A знает свой текущий cursor.
2. Реплика A отправляет изменения или запрашивает изменения после cursor.
3. Coordinator/backend принимает запрос, валидирует права и идемпотентность.
4. Coordinator/backend отвечает пачкой изменений или подтверждением.
5. Реплика A применяет новые изменения детерминированно.
6. Cursor обновляется только вперед.

## Идемпотентность

Повторное получение одного и того же event ID:
- не должно ломать состояние;
- не должно приводить к дублям;
- должно завершаться как no-op или подтвержденное повторное применение.

## Порядок

Полный глобальный порядок событий между всеми репликами не гарантируется.
Гарантируется только возможность:
- определить источник события;
- сравнить известный порядок внутри реплики;
- применить документированную conflict policy.

## Удаление

Удаление должно передаваться как доменный факт:
- либо отдельным событием `delete`;
- либо через tombstone;
- в любом случае получатель обязан уметь избежать "воскрешения" сущности
  устаревшей копией.

## Безопасность

Wire-level безопасность и аутентификация будут уточняться отдельно, но протокол
изначально должен допускать:
- идентификацию реплик;
- верификацию источника;
- ограничение неавторизованных sync-операций.

## Связь с HTTP API

HTTP API и sync-протокол решают разные задачи:
- HTTP API удобен для web-клиента и серверного CRUD;
- sync-протокол нужен для обмена изменениями между репликами.

На этапе MVP sync может быть транспортно выражен через HTTP, но нельзя сводить
всю sync-семантику к наивному REST CRUD без потери смысла.
