# Release stabilization Problem Ledger

## Purpose

Stable problem IDs prevent the project from rediscovering the same class of failure under a new stacktrace. Per-run machine-readable ledgers are generated by `release-gates`; this human ledger keeps the canonical family map and important accepted decisions.

## Canonical families

| ID | Layer | Meaning | Default severity |
|---|---|---|---|
| `REL-ENV` | Environment | Missing/unsupported local tools, shell, OS, path, permissions. | blocker if gate cannot run |
| `REL-DB` | PostgreSQL | Connectivity/auth/authority/migrations/disposable DB. | blocker for write-capable release gates |
| `REL-PROC` | Runtime | Managed process, port, PID, lifecycle, cleanup. | blocker for runtime gates |
| `REL-BE` | Backend | Rust/backend product or test failure after prerequisites. | blocker |
| `REL-FE` | Frontend | Build/unit/dependency failure after prerequisites. | blocker |
| `REL-BROWSER` | Browser tooling | Legacy browser automation prerequisite/install/cache. | infra blocker unless UI failure proven |
| `REL-UIUX` | UI evidence | Custom UI/UX Evidence Runner design/runtime/scenario failure. | blocker when mandatory UI evidence missing |
| `REL-SMOKE` | Smoke | API/UI smoke idempotency or write-safety issue. | blocker until isolated |
| `REL-DOC` | Docs/release | README/release notes/checklist/docs integrity. | blocker if release instructions are wrong |
| `REL-DEVCTL` | Patch conveyor | Patch/archive/check/push/transport issue. | blocker for patch acceptance |
| `REL-SEC` | Security/privacy | Secret leakage, unsafe artifact, redaction failure. | blocker |
| `REL-UNMAPPED` | Process | Unknown class awaiting classifier/decision. | blocker until mapped or accepted |

## Rules

- New recurring failure gets an ID before product code is patched.
- `REL-UNMAPPED` is a temporary holding state, not an acceptable final classification.
- Every blocker needs owner layer, evidence path and next probe.
- Accepted gaps must state why they are non-blocking and when they are revisited.
- Generated evidence stays in `.dev-bootstrap/runs` bundles, not in source archives.
