# Accelerated development strategy v1

## Назначение

Этот документ фиксирует рабочую стратегию ускоренной разработки `p2p_planner` после beta-prerelease. Его задача — не заменить архитектурные документы, а задать практический режим разработки, который помогает быстрее двигаться к v1 release без потери качества, контекста и состояния между чатами, локальными машинами, архивами и GitHub.

Стратегия исходит из текущей модели работы:

- разработка ведется крупными assistant-generated patch zip;
- патчи должны иметь `manifest.json`;
- локальный конвейер `devctl` применяет патч, делает архивы, запускает checks, пишет report, коммитит и пушит только при успешном результате;
- GitHub после успешного `devctl start` считается практическим source of truth;
- локальные архивы и reports нужны для диагностики, восстановления и передачи контекста.

## Главный принцип

```text
Крупные осмысленные патчи + короткий проверяемый производственный цикл.
```

Проект не переходит к микрозадачам ради самих микрозадач. Сильная сторона текущего процесса — крупные контекстные патчи, где несколько файлов меняются как единое решение. Но каждый такой патч должен заканчиваться чистым и проверяемым состоянием:

1. patch найден;
2. окружение и Git-состояние проверены;
3. pre-archive создан;
4. patch безопасно применен;
5. checks выполнены;
6. post/failed archive создан;
7. report записан;
8. commit создан только при успехе;
9. push выполнен только после commit;
10. следующий шаг понятен.

Ускорение достигается не за счет отказа от качества, а за счет удаления ручной рутины, повторных реконструкций контекста и потерь состояния.

## Что именно ускоряем

Ускоряется не только написание кода. Ускоряется весь цикл:

```text
вспомнить контекст -> выбрать полезный срез -> получить patch -> применить -> проверить -> зафиксировать -> продолжить на любой машине
```

Главные источники потерь, которые стратегия должна уменьшить:

- каждый новый чат требует заново собирать контекст;
- ручная архивация проекта легко ошибочна;
- легко забыть commit или push;
- разные локальные машины могут разойтись;
- patch zip без manifest не умеет выражать удаления;
- проверки могут требовать разные инструменты на Windows/Linux;
- после failed patch важно сохранить логи и состояние, но не делать скрытый rollback;
- большие документы сложно перечитывать перед каждой итерацией.

## Единицы работы

### Неправильная единица работы

```text
Сделать frontend.
Сделать sync.
Сделать mobile.
Поправить всё по release.
```

Такая постановка слишком широкая, плохо проверяется и легко размывает фокус.

### Правильная единица работы

```text
Сделать manifest-patch, после которого пользователь может проверить конкретное улучшение или проект получает конкретный release/операционный выигрыш.
```

Примеры правильных единиц:

- пользователь может пройти login -> workspace -> board -> card без legacy dev-header;
- разработчик может одним `devctl start` применить patch и получить commit/push;
- release gate v1 зафиксирован в одном документе с критериями stop/go;
- packaging spike показал, какой путь к exe/AppImage реалистичен;
- export/import flow работает на одном минимальном сценарии.

## Вертикальный срез

Вертикальный срез в этом проекте означает не обязательно маленькую задачу, а задачу с ощутимым итогом.

Срез может быть крупным, если он:

- имеет пользовательский, release или операционный смысл;
- меняет согласованный набор файлов;
- поставляется одним manifest-patch;
- имеет checks;
- может быть принят или отклонен по report/logs;
- не требует заново обсуждать всю архитектуру проекта.

## Режим общения с ассистентом

### Больше не обязательно: один чат = один этап

Старый режим был полезен для архитектурной разведки. Он помог разложить проект по областям: backend, frontend, local-first, sync, conflict resolution, p2p, security, testing, deployment.

После beta-prerelease основной режим меняется:

```text
один заход = один крупный проверяемый patch
```

Чат может обсуждать несколько связанных аспектов, если итогом становится один понятный patch или документированное решение. Главное — не плодить отдельные этапы ради структуры, если это увеличивает перенос контекста.

### Как формулировать запрос ассистенту

Хорошая постановка:

```text
Сделай manifest-patch для <цель>.
Контекст: <что важно>.
Ожидаемый результат: <что должно появиться>.
Checks: <что должно прогоняться>.
Не трогать: <границы>.
```

Плохая постановка:

```text
Давай подумаем обо всем проекте и что-нибудь улучшим.
```

Если нужна предварительная разведка, она должна заканчиваться коротким решением: делать patch сейчас, отложить в parking lot или разбить на два patch.

## Производственный конвейер

`devctl` — центральный инструмент стратегии. Он не является универсальным release manager. Его задача уже и полезнее:

```text
take latest assistant patch zip
validate environment and Git state
backup current project
apply patch safely
run checks
archive result
commit
push
record what happened
```

Базовый workspace:

```text
p2p_workspace/
  patches/
  archives/
  p2p_planner/
    backend/
    frontend/
    docs/
    tools/
      devctl.py
```

Основные команды:

```bash
python tools/devctl.py status
python tools/devctl.py start
```

`status` ничего не меняет. Он отвечает на вопрос: можно ли безопасно продолжать работу с этой машины.

`start` запускает производственный цикл: берет последний unapplied patch, проверяет условия, делает backup, применяет patch, запускает checks, пишет report, архивирует, коммитит и пушит только при успехе.

## GitHub как source of truth

После успешного `devctl start` GitHub должен быть актуален. Это снижает риск ситуации, когда на машине A есть одно состояние, на машине B другое, а в remote третье.

Правило:

```text
Успешная сессия не считается завершенной, пока изменения не запушены.
```

Локальный `.devctl/state.json` полезен, но не должен быть единственным источником правды. Успешные commits должны содержать trailers:

```text
Patch-Id: ...
Patch-SHA256: ...
Devctl-Version: ...
```

Так другая машина сможет понять историю patch даже без локального state registry.

## Документация

Документация должна быть минимальной, но операционной.

### Обязательные живые документы

- `docs/project-state.md` — короткий state-срез проекта.
- `docs/dev-log.md` — человеческий журнал важных итераций.
- `docs/development/accelerated-development-strategy-v1.md` — этот документ.
- `docs/release/release-v1-gate.md` — критерии готовности v1 release, когда будет создан.

### Правило для docs

Документ должен отвечать хотя бы на один вопрос:

- что сейчас актуально;
- что делать дальше;
- что нельзя потерять;
- как проверить результат;
- какое решение уже принято;
- почему не делаем опасный вариант.

Если документ не помогает действовать, он уходит в reference или parking lot.

## Project state

`project-state.md` — панель приборов, а не роман.

Он должен содержать:

- текущую фазу;
- актуальную ветку;
- последний стабильный commit/archive;
- что готово;
- текущий фокус;
- active risks;
- next best action;
- parking lot.

Его задача — быстро вернуть человека и ассистента в контекст без перечитывания всех больших документов.

## Dev log

`dev-log.md` не заменяет Git history. Он объясняет человеческим языком:

- какая задача решалась;
- какой patch применялся;
- какие checks прошли;
- где logs/report;
- какой результат;
- что делать дальше.

В v0 `devctl` не должен самовольно редактировать `dev-log.md`; это можно добавить позже отдельной осознанной функцией.

## Manifest-patch как контракт

Каждый patch должен иметь `manifest.json`. Manifest нужен не только для checks, но и для безопасного применения.

Минимально manifest должен уметь описывать:

- `patchId`;
- human-readable title/summary;
- `createdAt`;
- `apply.filesRoot`;
- `apply.delete`;
- `checks`;
- commit message;
- push policy;
- archive slug.

Особенно важно: удаления файлов и каталогов должны быть выражены явно через manifest, потому что zip с измененными файлами сам по себе не говорит, что нужно удалить.

## Checks policy

Checks задаются в manifest и являются частью patch contract.

В v0:

- `devctl` не устанавливает зависимости автоматически;
- `requiredCommands` проверяются до применения patch;
- checks запускаются последовательно;
- stdout/stderr пишутся в logs;
- commit/push возможны только после успешных checks;
- если checks изменили рабочее дерево, report должен это подсветить.

Будущие версии могут добавить:

- `setup` для explicit environment preparation;
- `services` для auto-start backend/frontend;
- readiness checks;
- browser smoke с логами backend/frontend;
- более тонкую commit include/exclude policy.

## Ошибки и recovery

Автоматический rollback в v0 не используется.

При failed checks:

- commit не создается;
- push не выполняется;
- failed-state archive создается;
- report записывается;
- рабочее дерево остается dirty для анализа;
- report должен дать ручную recovery-инструкцию.

Такой подход сохраняет максимум информации для диагностики и не скрывает опасных действий за магией.

## Мотивационная часть стратегии

Стратегия должна снижать трение входа в проект.

Для этого:

- стартовая команда должна быть очевидной: `python tools/devctl.py status`;
- рабочая команда должна быть одной: `python tools/devctl.py start`;
- после каждого успешного захода должен быть commit/push;
- следующий шаг должен быть записан явно;
- задачи должны давать ощутимый результат;
- не надо вручную чистить `target`, `node_modules`, архивировать проект и вспоминать, был ли push.

Цель — сделать возвращение к проекту дешевым:

```text
зашел -> status -> положил patch -> start -> получил результат
```

## Release-first фокус

После beta-prerelease главный фокус — не расширять бесконечно концепцию, а довести проект до v1 release.

v1 release не должен блокироваться:

- полным mobile app;
- полноценным production p2p;
- идеальным desktop packaging;
- всеми интеграциями;
- каждым возможным UI-polish.

v1 release должен блокироваться:

- сломанным core CRUD;
- сломанным web happy-path;
- невозможностью локального запуска;
- рисками потери данных в нормальном сценарии;
- неясной release-инструкцией;
- неуправляемым drift между машинами;
- отсутствием минимальных checks.

## Parking lot

Идеи, которые важны, но не должны сбивать v1:

- mobile architecture;
- apk;
- exe/AppImage;
- production p2p hardening;
- relay deployment;
- rich integrations;
- full auto setup;
- service orchestration;
- advanced devctl release management.

Они должны фиксироваться, но не становиться текущим фокусом без явного решения.

## Рекомендуемый порядок ближайших крупных patch

1. `devctl` stabilization.
2. `project-state.md` и `dev-log.md` в реальном проекте.
3. `release-v1-gate.md`.
4. Legacy cleanup: `X-User-Id`, auth/docs drift, старые deployment assumptions.
5. Release stabilization: checks, smoke, setup docs.
6. Packaging spike: self-host/dev bundle, потом desktop candidates.
7. v2/mobile/cross-platform parking lot refinement.

## Критерии хорошего patch

Хороший patch:

- имеет понятный пользовательский, release или operational goal;
- содержит manifest;
- не требует от пользователя помнить скрытые ручные шаги;
- проверяется через checks;
- при успехе приводит к clean pushed state;
- при провале дает failed archive и logs;
- не смешивает unrelated topics;
- не делает архитектурный разворот без документации.

## Критерии плохого patch

Плохой patch:

- меняет слишком много несвязанных областей;
- требует ручных действий, не описанных в manifest/README;
- не имеет checks;
- удаляет файлы без manifest delete;
- скрыто меняет окружение;
- автогенерирует неочевидные файлы;
- ломает возможность продолжить с другой машины;
- не оставляет понятного next step.

## Короткая формула

```text
Не мельчить задачи до бессмысленных микроправок.
Не раздувать задачи до бесконечных этапов.
Делать крупные осмысленные patch, которые проходят через один безопасный конвейер и оставляют проект в clean pushed state.
```
