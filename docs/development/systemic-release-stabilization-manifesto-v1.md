# Манифест системной стабилизации релиза v1

- Статус: Draft v1
- Дата: 2026-05-27
- Назначение: зафиксировать начало отдельной вехи, в которой проект перестает лечить каждый новый запуск точечным псевдофиксом и переходит к комплексному исследованию причин, из-за которых release/dev bootstrap цикл не становится устойчивым.

---

## 1. Зачем этот документ существует

Проект дошел до опасного, но полезного момента: почти вся энергия последних итераций уходит не на продуктовый прогресс, а на попытку наконец-то стабильно завести релизный цикл.

Текущий режим выглядит так:

```text
попытаться завести
→ получить ошибку и логи
→ сделать точечный fix
→ закоммитить fix
→ попробовать снова
→ вскрыть следующую ошибку
→ повторить цикл
```

Снаружи такой цикл похож на движение: коммиты появляются, отдельные ошибки исчезают, в логах становится чуть меньше старых проблем. Но по сути это уже не нормальная разработка, а симптом того, что проекту не хватает системной диагностической рамки.

Этот документ объявляет новую веху: **стабилизация release/dev lifecycle как отдельная инженерная проблема**, а не побочный шум вокруг “настоящей” разработки.

---

## 2. Главная формула

```text
Мы больше не чиним только последний stacktrace.
Мы исследуем систему, которая этот stacktrace порождает.
```

Ошибка запуска не является врагом. Она является наблюдением. Проблема начинается тогда, когда каждое наблюдение немедленно превращается в изолированный patch без общей гипотезы, без карты зависимостей и без ответа на вопрос: “почему мы не увидели это раньше?”

---

## 3. Что пошло не так в текущем цикле

Проблема не в том, что отдельные исправления были бесполезны. Наоборот, многие из них действительно закрывали реальные дефекты: Windows wrappers, npm launch, Vite startup, managed DB, CORS, smoke timing, test database credentials.

Проблема в другом: **каждый фикс открывал следующий слой неопределенности**, потому что релизный цикл проверялся как черный ящик.

Антипаттерны, которые теперь нужно явно остановить:

1. **Fix-after-fail loop** — исправляем только то, что упало последним.
2. **Commit stream as debugging diary** — Git-история превращается в журнал попыток, а не в последовательность осмысленных изменений.
3. **Implicit environment assumptions** — Windows/Linux, shell/cmd, npm wrappers, ports, PostgreSQL, env и старые процессы всплывают только после фактического падения.
4. **Late failure discovery** — проблемы обнаруживаются слишком поздно: уже после запуска длинного конвейера, а не на стадии дешевой диагностики.
5. **Unclear ownership of failure** — непонятно, ошибка в продукте, devbootstrap, smoke, локальном окружении, документации или в самом контракте release gates.
6. **Patch as reflex** — патч становится рефлексом на лог, а не результатом проверенной гипотезы.

---

## 4. Новая позиция проекта

С этого момента release/dev lifecycle считается полноценной частью продукта.

Это означает:

- bootstrapper — не вспомогательная “обвязка”, а важный developer-facing продукт;
- логи — не мусор после падения, а диагностический интерфейс;
- smoke/release gates — не формальность, а контракт доверия;
- ошибки окружения — не “у пользователя что-то не так”, а ожидаемый класс входных состояний;
- Windows quirks — не исключение, а часть поддерживаемой реальности;
- повторяемость запуска — не приятный бонус, а обязательное свойство v1.

Если проект нельзя стабильно поднять, проверить и остановить, значит v1 еще не готов независимо от того, сколько фич уже написано.

---

## 5. Принципы новой стратегии

### 5.1. Сначала гипотеза, потом патч

Перед исправлением нужно ответить:

```text
что именно сломано;
где граница проблемы;
как это воспроизводится;
какой класс похожих проблем закрывает исправление;
какая проверка докажет, что мы закрыли не только частный симптом.
```

Если ответов нет, это не fix-задача, а diagnostic spike.

### 5.2. Исправляем классы отказов, а не отдельные сообщения об ошибке

Плохая цель:

```text
починить текущий npm error
```

Хорошая цель:

```text
сделать Windows frontend launch предсказуемым для cmd/npm/vite wrapper cases и добавить раннюю диагностику выбранного launcher path
```

Одна ошибка должна приводить к улучшению всей диагностической карты.

### 5.3. Любой новый fix обязан улучшать наблюдаемость

Если патч меняет поведение запуска, он должен по возможности улучшать хотя бы один из пунктов:

- сообщение об ошибке;
- preflight check;
- classification;
- report artifact;
- smoke log;
- remediation hint;
- документацию по известному failure mode.

Иначе следующий сбой снова будет выглядеть как “что-то опять не стартует”.

### 5.4. Не смешиваем продуктовые баги и bootstrapper-баги

В одном отчете могут одновременно жить:

- дефект backend-кода;
- дефект frontend-кода;
- дефект smoke-сценария;
- дефект devbootstrap;
- дефект локального окружения;
- дефект документации.

Новая стратегия должна классифицировать их отдельно. Иначе есть риск править backend там, где сломан launcher, или переписывать smoke там, где не подготовлена БД.

### 5.5. Git-история должна снова стать историей решений

Коммит вида `fix: launch vite directly on windows` может быть полезным, если он закрывает понятный класс проблемы. Но серия из множества похожих fix-коммитов подряд сигнализирует, что мы используем Git как ленту экспериментов.

Новая цель:

```text
меньше реактивных псевдофиксов;
больше осмысленных пакетов стабилизации с диагностикой, критериями и объяснением.
```

### 5.6. “Запустилось один раз” больше не равно “готово”

Успех одного ручного запуска — это наблюдение, но не доказательство устойчивости.

Минимальный критерий устойчивости:

- запуск воспроизводится после clean checkout / clean archive;
- повторный запуск идемпотентен;
- остановка освобождает процессы и порты;
- повторный запуск после stop не зависит от случайного старого процесса;
- failures классифицируются до того, как пользователь начинает гадать.

---

## 6. Что должна дать следующая стратегия

Следующим шагом после этого манифеста должна стать не очередная точечная правка, а план комплексного исследования. Он должен описать:

1. **Карту release/dev lifecycle** — какие шаги реально выполняются от `diagnose` до `up`, `smoke`, `release-gates` и `stop`.
2. **Failure taxonomy** — какие классы отказов уже наблюдались и какие еще вероятны.
3. **Матрицу окружений** — Windows cmd/PowerShell, Linux shell, WSL, Node/npm versions, PostgreSQL modes, Docker availability.
4. **Контракт логов и артефактов** — что каждый шаг обязан оставить после себя, чтобы следующий анализ не начинался с нуля.
5. **Preflight before action** — какие ошибки можно обнаружить до запуска backend/frontend.
6. **Reproduction protocol** — как воспроизводить сбой на временном workspace без загрязнения основной истории.
7. **Patch acceptance criteria** — когда исправление считается доказанным, а когда это только эксперимент.
8. **Release gate stop/go policy** — какие падения блокируют v1, какие являются предупреждениями, какие уходят в backlog.
9. **Документированный каталог известных ловушек** — каждая однажды найденная ловушка должна становиться знанием проекта.

---

## 7. Что мы защищаем

Эта стратегия нужна не ради красоты процесса. Она защищает:

- время разработки;
- доверие к `devbootstrap`;
- читаемость Git-истории;
- переносимость проекта между машинами;
- способность ассистенту анализировать логи без потери контекста;
- будущий пользовательский опыт установки и запуска;
- уверенность, что v1 можно будет не только собрать, но и реально использовать.

Главное: она защищает моральное состояние проекта. Бесконечный цикл “ну теперь-то точно заведется” выжигает внимание и превращает разработку в угадайку. Хорошая инженерная стратегия возвращает ощущение контроля.

---

## 8. Объявление вехи

С этого момента в проекте открывается веха:

```text
Systemic release stabilization and diagnostic strategy
```

Цель вехи — превратить текущий хаотичный цикл запусков в понятный, наблюдаемый и воспроизводимый процесс.

Эта веха не отменяет продуктовую разработку, но временно ставит над ней важный вопрос:

```text
Можем ли мы доверять пути от исходников до работающего приложения?
```

Пока ответ не является уверенным “да”, любые новые фичи будут строиться на зыбком основании.
