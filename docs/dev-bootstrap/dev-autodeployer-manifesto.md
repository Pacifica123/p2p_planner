# Manifesto: зачем проекту нужен кастомный dev auto-bootstrapper

- Статус: Draft v1
- Дата: 2026-05-13
- Назначение: зафиксировать философию и мотивацию будущего инструмента, который автоматизирует локальное разворачивание, проверку и остановку dev-среды P2P Planner.

---

## 1. Короткая формула

Проекту нужен не просто набор `README`-команд и не один большой shell-скрипт, а **проектный авторазвертыватель**, который понимает текущую архитектуру P2P Planner и помогает стабильно пройти путь:

```text
скачан архив проекта
→ проверены инструменты и ОС
→ подготовлены env-файлы
→ найдена или поднята PostgreSQL
→ проверена правильная база
→ проверен backend
→ установлен frontend
→ подняты процессы
→ пройдены smoke/browser checks
→ после работы корректно освобождены процессы, порты и временные ресурсы
```

Главная цель — убрать ручную рутину и случайные ошибки, но не спрятать реальность за магией.

---

## 2. Почему обычного README уже мало

На раннем этапе было достаточно инструкции вида:

```bash
cd backend
cargo run
```

```bash
cd frontend
npm run dev
```

Но проект уже вышел из состояния skeleton. Сейчас в dev-запуске участвуют:

- Rust backend;
- PostgreSQL;
- миграции;
- auth/session конфиг;
- frontend на Vite;
- npm dependencies;
- browser smoke;
- Python smoke;
- env-контракты backend/frontend;
- dev-only режимы вроде `X-User-Id` и CORS allowlist;
- patch/devctl workflow;
- разные машины, ОС и состояния локального окружения.

Проблема не в том, что команд много. Проблема в том, что **каждая команда имеет скрытые предположения**.

Например:

- `cargo check` может пройти, но backend потом упадет на подключении к БД;
- PostgreSQL может быть установлен, но слушать другой порт;
- база может существовать, но быть не той;
- `.env` может быть создан, но указывать на старый инстанс;
- frontend может запускаться, но ходить не в тот backend;
- порт может быть занят старым процессом;
- браузер может показывать старое session/local storage состояние;
- smoke может падать не из-за кода, а из-за грязной dev-БД;
- Windows и Linux могут по-разному обрабатывать shell-команды, пути, кодировку и завершение процессов.

Авторазвертыватель нужен как слой, который эти предположения **проверяет явно**.

---

## 3. Чем этот инструмент не должен быть

Он не должен быть:

1. **Магическим инсталлятором всего подряд.**
   Если на машине нет Rust, Node или Docker, инструмент должен ясно сказать, чего не хватает, где это видно и что он может или не может сделать автоматически.

2. **Production deployment system.**
   Его первая задача — local/shared dev и проверяемый запуск проекта, а не Kubernetes, cloud provisioning и secret rotation.

3. **Скриптом, который молча чинит всё за пользователя.**
   Деструктивные действия вроде удаления базы, остановки чужого процесса, очистки Docker volume или перезаписи `.env` требуют явного подтверждения или отдельного режима.

4. **Заменой devctl patch conveyor.**
   Devctl применяет патчи и фиксирует состояние проекта. Auto-bootstrapper поднимает и проверяет runtime окружение. Эти инструменты могут интегрироваться, но у них разные зоны ответственности.

5. **Набором platform-specific костылей.**
   Внутри могут быть адаптеры под Windows/Linux, но пользовательский сценарий должен оставаться единым.

---

## 4. Главная философия

### 4.1. Inspect before act

Инструмент сначала диагностирует, потом действует.

Перед тем как запускать backend, он должен знать:

- есть ли `cargo`;
- какая версия Rust toolchain;
- существует ли backend `.env`;
- какой `DATABASE__URL` реально будет использован;
- доступна ли PostgreSQL;
- какая база указана;
- есть ли уже процесс на backend-порту;
- не является ли этот процесс старым backend из предыдущего запуска.

Перед тем как запускать frontend, он должен знать:

- есть ли Node/npm;
- совместима ли версия Node;
- установлены ли dependencies;
- совпадает ли `VITE_API_BASE_URL` с живым backend;
- не занят ли Vite-порт;
- не указывает ли frontend на другой backend.

### 4.2. Idempotence as default

Повторный запуск должен быть нормальным сценарием, а не лотереей.

Если `.env` уже есть, инструмент не должен без спроса перетирать его. Он должен:

- показать отличия от `.env.example`;
- подсветить отсутствующие ключи;
- предложить safe patch;
- сохранить backup перед изменением.

Если БД уже есть, инструмент не должен считать это ошибкой. Он должен:

- проверить, та ли это БД;
- проверить доступ;
- проверить состояние миграций;
- подсказать, чистая она или уже содержит данные;
- предложить отдельный test database для smoke.

### 4.3. Explainable automation

Каждый шаг должен отвечать на три вопроса:

```text
что проверяется
почему это важно
что делать, если не прошло
```

Плохой вывод:

```text
Backend failed.
```

Хороший вывод:

```text
Backend не смог подключиться к PostgreSQL.
DATABASE__URL указывает на 127.0.0.1:5432/p2p_planner,
порт 5432 открыт, но база p2p_planner отсутствует.
Можно создать базу автоматически или изменить backend/.env.
```

### 4.4. Cross-platform by design

Кроссплатформенность нельзя добавлять в конце. Она должна быть в фундаменте.

Инструмент должен учитывать:

- Windows PowerShell;
- Windows cmd quirks;
- Linux shells;
- WSL как отдельный пограничный случай;
- разные разделители путей;
- пробелы и кириллицу в путях;
- long paths на Windows;
- разные команды поиска и завершения процессов;
- разные способы проверки портов;
- разные сервисные модели PostgreSQL.

Практический вывод: ядро инструмента лучше делать на языке с хорошей стандартной библиотекой для процессов, путей и сетевых проверок. Для текущего проекта естественный baseline — **Python CLI**, потому что Python уже используется в smoke/devctl контексте и проще переносится между Windows/Linux, чем shell-only решение.

### 4.5. Safe by default, powerful by explicit mode

Baseline режим должен быть осторожным:

- не удалять данные;
- не убивать неизвестные процессы;
- не менять глобальный PostgreSQL;
- не перетирать `.env`;
- не делать destructive reset dev-БД;
- не чистить Docker volumes без подтверждения;
- не скрывать ошибки.

Более сильные действия допустимы только как отдельные команды:

```text
bootstrap diagnose
bootstrap prepare-env
bootstrap start-db
bootstrap start-backend
bootstrap start-frontend
bootstrap smoke
bootstrap stop
bootstrap reset-test-db --confirm
bootstrap clean-runtime --confirm
```

## 5. Что инструмент должен ценить больше всего

## 5.1. Воспроизводимость

Один и тот же архив проекта на двух машинах должен давать максимально похожий диагностический отчет:

- что найдено;
- чего не хватает;
- какие версии инструментов;
- какие env-файлы;
- какие порты заняты;
- какая БД используется;
- какие процессы запущены;
- какие проверки прошли.

Даже если автоматический запуск невозможен, диагностика должна быть полезной.

## 5.2. Наблюдаемость

Нужно сохранять логи:

- stdout/stderr backend;
- stdout/stderr frontend;
- вывод `cargo check` / `cargo test`;
- вывод `npm install` / `npm run build`;
- smoke reports;
- список PID процессов, поднятых инструментом;
- итоговый run report.

Идеальный результат локального запуска — не только “сервер поднят”, но и артефакт:

```text
.dev-bootstrap/runs/YYYYMMDD_HHMMSS/report.md
.dev-bootstrap/runs/YYYYMMDD_HHMMSS/backend.log
.dev-bootstrap/runs/YYYYMMDD_HHMMSS/frontend.log
.dev-bootstrap/runs/YYYYMMDD_HHMMSS/env-diff.md
```

## 5.3. Явное владение процессами

Инструмент должен различать:

- процессы, которые он сам запустил;
- процессы, которые уже существовали до него;
- процессы с похожими портами, но неизвестным происхождением.

Останавливать автоматически можно только то, что инструмент сам поднял и записал в state. Всё остальное — только после диагностики и подтверждения.

## 5.4. Compatibility with AI-assisted workflow

Проект разрабатывается итерациями через патчи. Поэтому авторазвертыватель должен помогать не только человеку, но и циклу “архив → патч → проверка”.

Он должен уметь:

- давать компактный machine-readable report;
- давать человекочитаемый summary;
- отделять проблему проекта от проблемы окружения;
- фиксировать, какие команды реально были выполнены;
- быть удобным как check в devctl manifest.

## 6. Приоритетные сценарии v1 инструмента

### Сценарий A: чистая машина, только архив проекта

Инструмент должен ответить:

- какие prerequisites отсутствуют;
- какие можно поставить вручную;
- какие env-файлы будут созданы;
- можно ли поднять PostgreSQL через Docker;
- какие команды будут выполнены дальше.

### Сценарий B: машина разработчика после перерыва

Инструмент должен найти:

- старые процессы;
- занятые порты;
- устаревшие dependencies;
- неправильный `DATABASE__URL`;
- миграционный drift;
- несовпадение frontend API URL и backend bind URL.

### Сценарий C: dev-БД уже существует, но “непонятно какая”

Инструмент должен проверить:

- имя базы;
- пользователя;
- доступность;
- наличие таблиц проекта;
- текущую версию миграций;
- есть ли данные;
- безопасно ли запускать smoke;
- нужна ли отдельная test-БД.

### Сценарий D: быстрый sanity перед патчем/коммитом

Инструмент должен выполнить минимальный gate:

- env sanity;
- PostgreSQL health;
- `cargo check`;
- backend smoke при поднятом backend;
- frontend dependency check;
- browser smoke при необходимости.

### Сценарий E: окончание dev-сессии

Инструмент должен помочь не забыть:

- остановить backend/frontend, которые он поднял;
- освободить порты;
- оставить PostgreSQL как есть или остановить compose-сервис;
- сохранить run report;
- явно предупредить о процессах, которые он не трогал.

---

## 7. Отношение к Docker

Docker полезен, но не должен становиться обязательной ширмой.

Baseline:

- PostgreSQL можно поднимать через Docker Compose;
- backend/frontend в dev лучше запускать нативно ради скорости и прозрачности;
- fully-containerized workflow можно добавить позже.

Инструмент должен уметь работать в двух режимах:

1. `native-dev`: Rust/Node запускаются на хосте, PostgreSQL локальный или Docker.
2. `compose-assisted`: PostgreSQL точно через compose, backend/frontend нативно.

Полная контейнеризация — future extension, а не обязательный фундамент.

---

## 8. Отношение к ошибкам

Ошибка — это не просто красный статус. Ошибка должна становиться диагностическим знанием.

Например, если backend не поднялся, отчет должен классифицировать вероятную причину:

- нет Rust/cargo;
- ошибка компиляции;
- отсутствует `.env`;
- не читается config;
- PostgreSQL недоступен;
- база отсутствует;
- миграции упали;
- порт занят;
- bind address недоступен;
- процесс поднялся и сразу завершился;
- health endpoint не ответил за таймаут.

Это будущая “база знаний” для инструмента и для документации.

---

## 9. Минимальный принцип проектирования

Каждая будущая команда auto-bootstrapper должна быть устроена так:

```text
preflight
→ plan
→ apply/start/check
→ verify
→ report
→ cleanup hint
```

Даже если в первой версии `plan` будет текстовым, он обязан быть. Пользователь должен видеть, что инструмент собирается сделать до того, как он начнет менять окружение.

---

## 10. Итог

Кастомный auto-bootstrapper нужен потому, что P2P Planner уже стал многослойным приложением, где локальный запуск зависит не от одной команды, а от согласованности ОС, инструментов, env, БД, миграций, backend, frontend, браузера и тестов.

Его философия:

```text
не магия, а проверяемая автоматизация;
не blind setup, а диагностируемый bootstrap;
не один скрипт для счастливого пути, а проводник по реальным сбоям dev-среды;
не замена человеку, а усилитель человека и AI-assisted patch workflow.
```
