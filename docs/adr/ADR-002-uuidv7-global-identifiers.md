# ADR-002: Глобальные идентификаторы UUIDv7 вместо централизованных serial ID

- Статус: Accepted
- Дата: 2026-04-01

## Контекст

Шаблонный проект использует обычный централизованный подход с автоинкрементными
числовыми идентификаторами. Для классического серверного CRUD это удобно, но для
local-first и p2p-синхронизации создает проблемы:
- коллизии между репликами;
- необходимость централизованного генератора ID;
- сложность офлайн-создания сущностей;
- неудобство слияния данных между устройствами.

## Решение

Для всех основных доменных сущностей использовать **UUIDv7** как базовый формат ID.

Под UUIDv7 в проекте понимается:
- глобально уникальный идентификатор;
- частично time-ordered поведение;
- пригодность для локальной генерации без обращения к серверу;
- удобство для индексации и сортировки по времени создания лучше, чем у UUIDv4.

## Сущности, на которые распространяется решение

UUIDv7 обязателен как минимум для:
- users
- devices
- workspaces
- workspace_members
- boards
- board_columns
- cards
- comments
- attachments
- checklists
- checklist_items
- board_labels
- custom_fields
- card_custom_field_values
- activity_log
- replicas
- change_events
- tombstones
- integration_connections
- integration_links

## Исключения

Чисто технические или внутренние sequence-поля допустимы только там, где они:
- не являются внешним идентификатором сущности;
- не используются для синхронизации между репликами;
- не публикуются в API как canonical ID.

Примером исключения могут быть внутренние version counters, ordinal position
или sequence для локальной нумерации событий внутри одной реплики.

## Следствия

### Плюсы
- Упрощается офлайн-создание сущностей.
- Снижается зависимость от централизованного сервера.
- Проще проектировать синхронизацию и импорт/экспорт.
- Улучшается масштабируемость на будущее.

### Минусы
- UUID тяжелее для ручного чтения и отладки.
- Нужно сразу выстроить единый формат во всех слоях: БД, backend, frontend, sync.

## Правила реализации

1. В OpenAPI идентификаторы описываются как `type: string`, `format: uuid`.
2. В БД canonical ID колонка хранится как `uuid`.
3. В DTO и JSON payload ID передаются как string UUID.
4. Нельзя смешивать разные стандарты ID для одной и той же доменной сущности.
