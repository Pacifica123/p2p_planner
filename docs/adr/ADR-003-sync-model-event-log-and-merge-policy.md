# ADR-003: Модель синхронизации через change log, tombstones и явную политику merge

- Статус: Accepted
- Дата: 2026-04-01

## Контекст

Обычная CRUD-модель предполагает, что сервер видит последовательность изменений
в реальном времени и может быть абсолютным источником истины. В local-first /
multi-replica системе это неверно:
- изменения могут создаваться локально;
- одна и та же сущность может редактироваться из разных клиентов/устройств;
- удаление должно корректно воспроизводиться на других репликах;
- синхронизация может быть частичной и повторной;
- не все конфликты можно решить одинаково.

При этом MVP проекта — это не full p2p protocol suite, а web-first продукт с
sync-ready foundation.

## Решение

Принимается следующая модель:

1. Каждое устройство или клиентская копия состояния рассматривается как **реплика**.
2. Изменения фиксируются как **change events** или эквивалентная change-based модель.
3. Удаление выражается через **tombstones** или эквивалентный доменный факт удаления.
4. Для конфликтов вводится **явная стратегия merge**, а не "как получится".
5. HTTP CRUD сохраняется как удобный прикладной интерфейс, но внутренняя модель
   проектируется с учетом change-log semantics.

## Важное уточнение по слоям

Нужно различать три уровня:

### 1. Доменная sync-семантика
Это базовые понятия:
- replica;
- change event;
- tombstone;
- cursor;
- conflict policy.

Они обязательны как архитектурный контракт.

### 2. MVP-реализация
В первом релизе достаточно:
- backend-coordinated sync для web-клиента;
- device/replica-aware модели;
- идемпотентного приема изменений;
- детерминированной conflict policy для core сущностей.

### 3. Post-MVP transport evolution
Отдельные peer/relay/snapshot transport flows могут развиваться позже.
Наличие `hello`, `snapshot_offer` или peer-to-peer обмена в документах не
означает, что это уже обязательная часть MVP.

## Базовые положения

### Реплика
Реплика — отдельный источник изменений, имеющий:
- replica ID;
- информацию об устройстве или клиентском контексте;
- свои счетчики/позиции синхронизации;
- свой change stream.

### Change event
Изменение должно содержать как минимум:
- event ID;
- replica ID;
- entity type;
- entity ID;
- operation;
- logical timestamp / lamport-like counter / device time metadata;
- payload patch или snapshot-представление поля;
- causation/correlation metadata при необходимости.

### Tombstone
Удаление сущности не должно сразу уничтожать возможность донести факт удаления
до других реплик. Поэтому удаление фиксируется отдельной записью или полем,
которое позволяет:
- синхронизировать удаление;
- избежать "воскрешения" сущности из старой копии;
- безопасно чистить данные позже через compaction/GC стратегию.

## Политика разрешения конфликтов

### По умолчанию
Для большинства простых scalar-полей используется **LWW per field** при условии,
что:
- поле не агрегирует список;
- потеря промежуточных состояний допустима;
- конфликт можно трактовать как выбор более нового значения.

### Для ordered-структур
Для колонок и карточек порядок — отдельная проблема. Поэтому:
- позиция хранится отдельно от ID;
- reorder должен быть отдельной операцией;
- нельзя сводить порядок только к `updated_at`.

### Для списков и коллекций
Коллекции не должны наивно перетираться целиком. Для них нужен merge на уровне:
- add/remove element;
- update element;
- move element.

### Для высокорисковых конфликтов
Для некоторых действий допустим ручной конфликт или защитный reject, если:
- автоматическое слияние разрушает смысл;
- требуется подтверждение пользователя;
- операция нарушает права доступа или уже отозванный контекст.

## Следствия

- В БД нужны или допускаются таблицы `replicas`, `change_events`, `sync_cursors`, `tombstones`.
- Сервисный слой должен уметь транслировать CRUD в change-aware модель.
- Conflict policy должна быть документирована и тестируема отдельно.
- Импорт/экспорт и backup позже должны учитывать event-log и snapshot model.
- Мы **не принимаем** на старте full event sourcing как обязательный стиль всей системы;
  речь идет о sync-aware модели, а не о догме реализации.
