# ADR-004: Modular monolith на старте, интеграции и внешние инструменты через adapters

- Статус: Accepted
- Дата: 2026-04-01

## Контекст

Шаблонный проект already показывает полезный паттерн:
- модульный backend;
- отдельные dto / handler / service / repo;
- OpenAPI как общий контракт;
- миграции как источник истины по БД.

Для нового проекта это полезно сохранить, но домен и границы модулей должны быть
другими. Также есть явное требование универсализации и будущей поддержки внешних
инструментов вроде Obsidian, GitHub и других систем.

## Решение

На старте проект строится как **modular monolith**:
- один backend deployment unit;
- один основной web-клиент;
- единая БД на первом этапе;
- четко разделенные модули по домену.

Интеграции с внешними системами строятся через **adapter/provider abstraction**,
а не через вшитую в core-логику реализацию под одну конкретную платформу.

## Что перенимаем из шаблонного проекта

1. Разделение backend по слоям:
   - dto
   - handler
   - service
   - repo

2. Отдельные SQL migrations как основной путь изменения БД.

3. Отдельный каталог `docs/api/` с OpenAPI-контрактом.

4. Наличие smoke-скриптов для API.

5. Feature-based подход на frontend.

## Что меняем по сравнению с шаблоном

### Смена архитектурной оси
Вместо role-centric домена проектируется workspace-centric домен:
- workspace
- board
- column
- card
- member
- appearance
- sync
- integrations

### Смена модели идентичности
Вместо упора на server-generated IDs — UUIDv7 и реплики.

### Смена модели удаления
Вместо простого hard delete — tombstone-aware deletion.

### Смена фокуса extensibility
Вместо логики "под одну задачу продукта" — универсализируемая система:
- произвольные доски;
- произвольные колонки;
- кастомизация внешнего вида;
- расширяемые интеграции;
- заготовки под импорт/экспорт и внешние провайдеры.

## Важное уточнение по стадиям

Архитектурные модули **не равны** обязательным физическим пакетам первого этапа.

Для MVP как отдельные зоны ответственности обязательны:
- auth / users;
- workspaces;
- boards;
- cards;
- labels;
- checklists;
- comments;
- sync.

Могут существовать как future-ready границы, но быть отложены или временно жить
внутри более крупных модулей:
- board_appearance;
- attachments;
- custom_fields;
- activity как отдельный пользовательский surface;
- integrations;
- invitations как отдельный модуль.

## Следствия

- Не нужно дробить систему на микросервисы до появления реальной потребности.
- Модульность должна быть сильной внутри монолита.
- Интерфейсы интеграций проектируются заранее.
- Будущие модули не должны раздувать MVP и обязательные first migrations.
- GitHub, Obsidian и подобные системы сначала появляются как stubs/providers,
  а не как hard dependency проекта.
