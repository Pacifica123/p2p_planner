# v1.0.0-beta.2 release artifact contract

- Release tag: `v1.0.0-beta.2`
- Release type: GitHub Pre-release / beta
- Scope: define the files that must be attached to the GitHub release and how to validate them before upload.
- This document describes release artifacts only. Devctl patches and project snapshots must not embed large binary payloads.

## Required GitHub assets

| Asset | Required | Purpose | Notes |
| --- | --- | --- | --- |
| `p2p-planner-v1.0.0-beta.2-web-win64.zip` | yes | Windows self-host bundle | Contains backend `.exe`, frontend `dist`, env examples, `docker-compose.dev.yml`, and `README_RELEASE.md`. |
| `p2p-planner-v1.0.0-beta.2-linux-x86_64.AppImage` | yes | Portable Linux launch artifact | One executable file for x86_64 Linux. See AppImage section. |

| Source code archives | automatic | GitHub-generated source | GitHub creates them from the tag. |

## Windows executable bundle

The Windows asset is not just a bare `.exe`. It should be a self-host bundle that a user can unzip and run locally.

Expected contents:

```text
p2p-planner-v1.0.0-beta.2-web-win64/
  README_RELEASE.md
  docker-compose.dev.yml
  backend/
    p2p-planner-backend.exe
    .env.example
  frontend/
    dist/
    .env.example
```

Validation before upload:

1. unzip the bundle into a clean folder;
2. start PostgreSQL with the bundled `docker-compose.dev.yml` or point `.env` to an existing local PostgreSQL;
3. run the backend executable;
4. serve the frontend `dist` on the documented port;
5. open the app, sign up/sign in and create a workspace, board, column and card;
6. record the exact artifact hash in `SHA256SUMS.txt`.

## Linux AppImage asset

AppImage is a portable Linux executable package: a single `.AppImage` file that mounts itself as a virtual filesystem and starts the bundled application without installing files into system directories.

For beta.2, the AppImage is a release artifact contract, not a source-level dependency. The project remains Rust/Axum backend + React/Vite frontend; it is not a React Native Linux app. The AppImage packaging must therefore make one explicit runtime decision:

- bundle the backend executable and static frontend assets with a small launcher that starts the local service and opens/serves the UI; or
- use a future desktop wrapper/runtime if that becomes the selected packaging architecture.

Minimum AppImage expectations:

- target `x86_64-linux-gnu` unless the release explicitly names another architecture;
- include application binaries, frontend resources and non-system shared libraries required by the bundled runtime;
- do not bundle system-critical libraries that should remain provided by the host, such as glibc/vdso;
- include `.desktop` metadata and icon metadata when available;
- run without root;
- avoid writing outside user-owned application/cache/config/data locations;
- document external requirements that are not bundled, especially PostgreSQL if the AppImage does not embed a database/runtime story.

Useful tooling candidates:

- `appimagetool` to build the final `.AppImage` from an AppDir;
- `linuxdeploy` to prepare an AppDir and collect shared-library dependencies;
- Tauri/Electron builders only if the project later adopts such a desktop wrapper.

Validation before upload:

1. make the file executable: `chmod +x p2p-planner-v1.0.0-beta.2-linux-x86_64.AppImage`;
2. run it as a normal user, not root;
3. confirm the main UI opens and the backend/runtime startup path is clear;
4. verify the core flow: sign up/sign in, workspace, board, column, card;
5. test at least one older Ubuntu-family environment plus one non-Ubuntu environment if available;
6. confirm missing PostgreSQL/runtime prerequisites are reported clearly if they are not bundled;
7. record the exact artifact hash in `SHA256SUMS.txt`.

## Size and safety notes

- AppImage can be large, especially if a browser/desktop engine is bundled. Strip symbols and remove unused build files where safe.
- Do not include `.env`, secrets, local databases, `node_modules`, `target`, logs or `.dev-bootstrap` runtime state in release assets.
- If signatures are published, document the public key or verification command in the release body.
- Do not treat a platform artifact as ready merely because source-level release-gates passed; every uploaded artifact needs at least one artifact-level smoke run.
